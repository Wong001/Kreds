# Kreds — Roadmap & Status

*Product name: Kreds (renamed from Loop). Package/module names are still `hearth`. Last updated: 2026-07-08.*

Kreds is a private, non-commercial, peer-to-peer social platform: friends-only (added via short-lived, single-use invite codes — no discovery, no lookup), data on hardware you own, no follower/like/view metrics by architecture. This doc is the single source of truth for what's built, what's honest about its limits, and what's next.

See `docs/superpowers/specs/` and `docs/superpowers/plans/` for the per-feature design + implementation history.

---

## Shipped (18 features)

Each was built spec → plan → task-by-task implementation → per-task review → whole-branch review, and merged only at "ready to merge."

1. **Vertical slice v0.1** — signed posts + photos with expiry, in-person QR friend-add, device enrollment/revocation, gossip sync over TCP, web UI. Content-addressed SHA-256 blobs, expiry/tombstone sweep, GC.
2. **Encrypted DMs v0.1** — end-to-end 1:1 messages (text + photos). Per-device X25519 keys announced via an `enckey` message; ChaCha20-Poly1305 sealed body + per-device key-wrap; strict no-relay routing (friends never carry your DMs); home-node store-and-forward; revocation self-logout. *First real confidentiality property.*
3. **Visual system + profiles v1** — "quiet canvas, loud people" theme, ring-o wordmark, Hearth→Loop rename; customizable profiles (bio, accent, avatar shape/size/placement, banner) as **structured options, never user CSS** (XSS-safe); Pillow image transcode gate on uploads.
4. **Stories v1** — 24h-ephemeral photo + short muted video. Video transcode gate (imageio-ffmpeg): ≤15s, ≤720p, audio stripped, H.264, bitrate-capped <5MB, poster frame; hardened (subprocess timeouts, `-nostdin`, protocol whitelist). Seen-state is **localStorage only** — no view/seen metrics leave the client (thesis).
5. **UI polish + dark mode** — neutral dark theme (persisted toggle); accent widened to any hex (16 swatches + custom picker) applied app-wide with luminance-derived contrast; Messages rebuilt as a real chat; styled controls + inline SVG icons; empty states; friend-flow declutter.
6. **Deletion hardening + honesty pass** — delete-before-content race fixed for every message kind (was posts-only; deleted DMs/stories/profiles could resurrect under out-of-order gossip), covering index, adverse-order convergence tests over real sockets; honest delete confirm in the UI; README/ROADMAP state the D3 boundary.
7. **DM forward secrecy v0.2 (slow ratchet)** — daily enckey rotation riding the existing latest-wins `enckey` messages (zero protocol change), 7-day grace then permanent key deletion, per-device encrypted `dm_keys` content-key cache so history survives rotation; the FS property is proven by test (a `keys.json`-only leak cannot decrypt pre-rotation envelopes). Double Ratchet / per-pair chains rejected: mailbox latency collapses their granularity to the window anyway (see spec).
8. **Tor transport v1** — nodes reachable as stable `.onion` services, dialing peers through a hand-rolled asyncio SOCKS5 dialer; Tor is invisible (managed binary: bundled-with-app → cached → pinned hash-verified download, headless on loopback); dual-stack onion-preferred with a privacy guardrail (never gossip a non-onion address for a peer with a known onion); 45s onion sync cadence; onion service detached so it outlives the publishing connection. Windows-x86_64 only for now. **Honest validation status:** the onion-gossip path was proven end-to-end by the feasibility spike (`docs/spikes/2026-07-03-tor-feasibility-spike.md`, 12/12 real syncs); the transport is unit-tested offline and a `TOR_E2E=1`-gated live test exists, but the post-spike detach fix has **not yet been re-confirmed by a green live run** (the sandbox's Tor bootstrap is currently blocked). Live re-validation on a cooperative network is the one open item.
9. **Kreds rename + v3 reskin** — product renamed Loop→Kreds throughout the UI (title, wordmark, copy, icons; no user-facing "Loop" remains); full web client rewrite to a journal-first IA — day-grouped feed with a "that's everything" end state, a chip-bar filter (Everyone / Inner kreds / per-person), the compact circle rail that expands into a radial overlay map, and profile modals (identity-color banner, ring status, move-between-rings) opened from anywhere a person appears; deterministic per-person identity colors (one derivation, shared by avatars/rings/map nodes); fonts self-hosted (no external font host). **Installable as a PWA** — manifest + a shell-caching service worker (app shell only; never caches `/api/` or `/ws`, scope covers `/`) — "Add to Home Screen" on iOS/Android, "Install" on desktop; the shell loads offline, data still requires the node. Two read-only API additions (`GET /api/kreds`, profile `ring`/`since`) back the map/chips/modal; no protocol or crypto change. Honesty guards held over from prior slices: no receipts "who holds this" popover; unread markers stay localStorage-only. Internal package/module names remain `hearth`. *(The profile modal named here is superseded by feature 10, below — kept as an accurate record of what shipped at the time. The "Add to Home Screen" on iOS claim is likewise superseded by feature 14's decision to drop PWA as the iOS strategy in favor of a native app — the manifest/service worker themselves are unchanged and still installable on desktop/Android.)*
10. **Kreds Me/profile Slice A** — the profile modal from feature 9 is retired; every person is now a full profile **page** (`#view-profile`), opened from anywhere a person appears (journal avatar/name, chip bar, friends list, circle map — including from inside the expanded circle overlay, which now closes itself on open so the page is never stranded behind it) and returning via a **Back** button to wherever you came from. The page shows a banner (identity-color fill, or an uploaded banner image), a customizable avatar (shape: circle/squircle/square/triangle; size; placement), ring status with a since-date, a **Message** action, and **move-between-rings**; self-editing is unified onto the same page (no more separate inline editor slot). The **Me view** is reorganized into a two-column layout: your own card in the main column, **Friends + Devices grouped in a right-hand side column** (heading relabeled "Friends"), with "Edit profile" navigating to your own profile page. This profile page is the foundation the later **block-based-profile** slice (grids/columns/custom content layout, feature still below) builds on; block *content* itself is not yet implemented. **Unfriend** (Slice B) shipped next, as feature 11 below. *(The separate Me view described here is superseded by the small IA refinement noted at the end of feature 12 — Friends/Devices now live in a self-only strip on the profile page itself, not their own view.)*
11. **Kreds unfriend (Slice B)** — remove a friend, signed-notice-driven and direct-only, never a block. `node.unfriend(id)` tears down the relationship immediately and locally (drops the friend, their content, the DM thread with them, any ring record about them) and queues a signed defriend notice minted through the same enrolled-device path as a revocation cert. Delivery (`deliver_defriends`) dials the target directly — never mesh/broadcast, nothing else is served on that connection — and is ack-based, not session-completion-based: a declined or tampered notice keeps the record queued for retry until the target's own applied-list names it or a 14-day window expires. On the recipient's side, an applied notice makes the device self-delete everything the remover authored, drop the remover from its identities, and mark them "no longer connected" in the UI; the same in-person invite ceremony used to become friends is the only way back, and re-adding clears both a stale outbox notice and the disconnected marker. **Resurrection is structurally blocked, not just discouraged**: proven by a live three-node integration test — A, B, C all mutual friends, A's post synced to both; A unfriends B and delivers; B purges A and forgets A; C (unaffected) still holds A's post and syncs with B for real, but B never re-acquires it (B no longer declares A as known, so the recipient-entitlement filter never even offers it, and B's own unknown-identity ingest gate would refuse it regardless). A companion test confirms the negative: a bare refusal/offline session — with no notice ever delivered — never purges anything, because no refusal-trigger code path exists. No block/ban primitive: honest limits are stated plainly (a modified client can retain what it already had, a screenshot survives everything, and a device offline for the full 14-day window never receives the notice and keeps the content until it does or until re-friending supersedes it). **Upgrade note:** this slice bumps the wire `PROTOCOL` to `hearth/v0.2` (the defriend phase is a new mandatory session phase), which by design invalidates any identity/cert/message persisted under `v0.1` — a hard breaking migration. There are no released versions or real users, so the only action is to re-initialize any local cast (delete `run/` for the demo); a `v0.1` store would otherwise surface as mystery "bad peer cert" refusals.

12. **Kreds curated profile (Wall MVP)** — every post now carries a `placement`: `"journal"` (default, unchanged behavior — home feed + a profile's Journal section) or `"profile"` (a profile's Wall only, never the home feed). Placement is orthogonal to scope: a profile post is still per-recipient encrypted and honors the Inner/Kreds selector exactly like a journal post — the only new field is `placement`, carried in the signed payload (`make_post`/`compose_post`/`validate_payload`; a missing field defaults to `"journal"` for backward compatibility with pre-existing posts). `node.feed()` now returns journal-placement posts only; `node.posts_by(identity_pub, placement=None)` gained an optional filter; `profile_view()` calls it twice and returns `wall` + `journal`, both already decrypt-filtered server-side (a viewer who can't decrypt a post sees it in neither section, not a crash). `/api/post` accepts the `placement` form field; `/api/profile/{id}` returns the split `wall`/`journal` in one fetch. **Every profile page now renders two sections** — Wall (that person's profile posts) and Journal (their journal posts the viewer can currently decrypt) — reusing the existing post-entry renderer for both, with independent honest empty states. **Self profile renders exactly as others see it**: the inline editor dump is gone; a small cogwheel button (self-only, also reachable from the Me card) opens the existing profile editor as an overlay on top of the page instead, and saving re-renders the page underneath before closing the overlay. A wall composer above your own Wall posts directly with `placement=profile` (default scope Kreds, Inner/Kreds selector, reusing the journal composer's visual pattern). Proven end-to-end by a live two-node integration test over real sync sockets (`tests/test_curated_profile_integration.py`): a journal post and a profile post both survive encrypt→sync→decrypt with placement intact — the profile post never reaches the recipient's feed or Journal section, the journal post never reaches the Wall — and an *expiring* journal post appears in Journal before it expires and never on the Wall (placement, not expiry, gates the Wall). Also confirmed with a real Chromium browser session (Playwright) against a live node on a free port: cogwheel opens the overlay, an uploaded avatar+banner render on the header itself (not just inside the editor), saving closes the overlay, a Wall-composer post lands on the Wall and not the home feed, and a normal journal post lands in the feed and the profile's Journal section and not the Wall. **Honest limit / immediate next**: this ships *placement*, not *layout* — the Wall is a plain reverse-chronological list, same rendering as the Journal, with no grids, custom arrangement, or pinning. 13. **App-lock (Windows desktop)** — a device-local screen lock, closing the "unlocked device" custody gap the windowed DM-forward-secrecy docs (feature 2) name. Pick a PIN or passphrase (`cred_type` is a UI-only distinction — numeric keypad vs. text field; the crypto path is identical either way). At rest, the device's full secret bundle (`device_priv`, `identity_priv`, `enc_priv` + `retired_enc`, `storage_key`) is sealed into `applock.json` under a master key derived by combining `scrypt(credential)` with a random 32-byte device secret via HKDF — that device secret itself exists only sealed behind **Windows DPAPI** (tied to the Windows login), so a short PIN is safe at rest: the device supplies the entropy, the credential is just what you type. `keys.json` keeps only the non-secret fields plus an `applock: true` marker once enabled. The node **boots locked** — `DeviceKeys.locked_from_json` holds no private material, and `sign_message`/`sign_raw` raise rather than silently no-op — and the API layer's `locked_gate` middleware returns **423** for every `/api/*` route except an exact-path allowlist (`/api/unlock`, `/api/applock` status; `/api/applock/settings` is a different path and stays gated). `node.enable_applock`/`unlock`/`lock`, plus `POST /api/unlock` / `/api/lock` / `/api/applock/setup` / `/settings` / `/change` / `/disable`; a wrong credential escalates an in-memory throttle (0 → 5s → 30s → 300s as failures mount, reset on success or process restart) — never an auto-wipe. Auto-lock is **node-tracked, not browser-tracked**: the node's own periodic loop locks on idle (configurable minutes, off by default) or a wall-clock jump consistent with the process having been suspended and resumed — HTTP request timing isn't a reliable sleep-detector. Client: a full-screen lock overlay (PIN keypad or passphrase field per `cred_type`, live throttle countdown) hooks the global `fetch` wrapper so *any* 423 anywhere in the app surfaces it, hiding the whole app subtree (not just the visible view) until unlock; a self-only Settings panel covers enable / idle timeout / lock-on-sleep / change credential / disable / lock-now. Proven end-to-end by a real-DPAPI integration test (`tests/test_applock_integration.py`): create a node, enable App-lock, confirm `keys.json` carries none of `DeviceKeys.SECRET_FIELDS`, reboot it for real (`HearthNode(dir)` loading fresh from disk, not an in-memory enable+lock shortcut), confirm it boots locked and a content op raises, drive 423 → unlock → 200 over the real HTTP API, post a profile Wall block, lock and unlock again through the API, and confirm the block still decrypts — the secret bundle, including the content-key cache's `storage_key`, round-trips intact through a genuine DPAPI-seal/unseal + scrypt re-derivation — plus a second test pinning the same 5s/30s/300s throttle tiers as observed in the real `/api/unlock` response body (not just at the node-method level, where `test_applock_api.py` already pins them). Full suite green (397 passed, 1 pre-existing skip, confirmed clean across 2 consecutive runs) plus `node --check` clean; the non-App-lock demo path and device-revocation tests are unaffected (12/12 green in that slice). **Honest threat model, stated plainly:** protects a device idle, asleep, or stolen while off, and a stolen `keys.json` + `applock.json` pair on its own — without also having the Windows login that seals the device secret, DPAPI cannot unseal it. Does **not** protect against malware or a shoulder-surfer while the app is genuinely unlocked and running — a different boundary (revoke the device instead). OS-suspend detection is a wall-clock-jump heuristic, not a real OS suspend/resume hook, pending a packaged desktop wrapper that can subscribe to that event directly. **Windows-only for now** — DPAPI is a Windows API and no off-Windows fallback has been built, so the feature is simply unavailable, not weaker, elsewhere. Deferred: a pattern-style credential, biometric unlock, true OS-suspend hooks, cross-device settings sync. **Client lock-screen/settings behavior is verified by the user (August) per the project's testing-workflow split** — Playwright was not run for this feature; a verification checklist is handed over at merge.

16. **Auto-update Phase 2a (signed manifest + web hot-update + core staging)** — the mechanism for shipping Kreds updates, wired through from source; no distributable `.exe` exists yet (Phase 2b, see "Next up"). A one-time offline **Ed25519 release keypair**: the app bakes in only the **public** key (`hearth/update.RELEASE_PUBKEY`); the private key is gitignored (`release_private_key*`) and lives offline with the developer, never in this repo — verified directly (`tests/test_update_trust.py::test_release_pubkey_present_and_no_private_committed` greps `git ls-files` for it). A release is one signed `manifest.json` (+ sibling `manifest.sig`): `version`, `min_core_for_web`, and a `web` and/or `core` bundle entry, each `{url, sha256, size}` — signing the manifest authenticates the bundles too, since the manifest pins their hashes. `update.check()` fetches + verifies against `RELEASE_PUBKEY` and **never raises to its caller** — a bad signature, tampered manifest, fetch failure, or a version that isn't strictly newer than `hearth.__version__` (no downgrades) all resolve to "no update," not a crash or a 500. `apply_web(info, web_dir)` downloads a "web" bundle (a deterministic zip of `hearth/web/`, `update.build_web_bundle`), re-verifies its sha256, and atomically swaps it into `web_dir` with rollback (rename to `.web-bak`, rename `.web-new` into place; any failure restores `.web-bak` so the served dir is never left missing/half-written) — targeting whichever directory `build_app(node, web_dir=...)` is *actually* serving, not a hardcoded path. `stage_core(info, staging_dir)` downloads + hash-verifies a "core" bundle but only stages it (`update-staging/pending-core.zip` + a version marker); nothing about the running process changes — *applying* a staged core update on restart is Phase 2b, not built yet. New `GET /api/update/check` / `POST /api/update/apply` (both inside `build_app`, closing over the real served `wd`) and a self-only Updates panel in Settings (`renderUpdateSettings()`, manual check/apply only, no background auto-check loop). The feed is modeled as GitHub Releases assets (`FEED_URL`), overridable via `HEARTH_UPDATE_FEED` for dev/tests; every test in the slice signs its own **throwaway** key and points at a local `file://` feed, never the real release key. **A real Windows bug found and fixed by the Task 4 integration test, not papered over**: `apply_web`'s `os.rename` of the served directory reliably failed with `PermissionError [WinError 5]` when that directory was under genuine concurrent HTTP load (confirmed reproducible across repeated runs) — Windows transiently refuses to rename a directory while any file inside it still has an open handle, which a request actively being served out of that exact directory can briefly hold. Fixed with a short bounded retry around every rename (`_rename_with_retry`, ~3s budget) that rides out the transient window without weakening the rollback guarantee; proven by two new tests — a deterministic reproduction (an explicitly held-open file handle released partway through the retry budget) and a realistic concurrent-HTTP-load version through the real `build_app`/`TestClient` path. The staged marker's `version` string — used as a `versions/<version>` path segment that is later **executed** — is validated as a plain dotted-numeric string (`is_valid_version`) before any of that, closing a path-traversal-to-arbitrary-execution primitive; `packaging/launcher.py` independently re-validates `current`'s content the same way, defense-in-depth, right before joining it into a path and running it. A good bundle always extracts into a **brand-new** `versions/<new>/` directory (a pre-existing directory of that name is never trusted by bare existence, even if a file named `Kreds.exe` is already sitting there — it's cleared and re-populated with the just-verified bytes) and only then flips `current`, keeping the prior version in `previous` for rollback if the new one crashes within a short window of starting. The shell's "Restart to finish" action (`Api.restart()`, offered after a staged core update) spawns this stable launcher **detached** and quits the current instance; the launcher applies the update on its own fresh start. **Publish pipeline:** `hearth/cli.py` gains `release-build` (assembles `web-<ver>.zip` + `core-<ver>.zip` + a `manifest.json` pinning their real sha256/size), `release-sign` (signs it with the offline release private key — a manual, local, never-CI step), and `release-publish` (a thin `gh release create` wrapper); ## Honest status — what's demo vs. real

A developer reading this repo should know the current boundaries:

- **Transport: Tor is wired; plaintext TCP is the demo default.** Nodes run as onion services (`demo --tor`) with encrypted, endpoint-authenticated, metadata-blind transport; the localhost demo stays plaintext TCP for speed. Onion identity is stable across restarts; first sync can take ~a minute (onion peers sync on a slower interval). *Live re-validation of the post-spike code is still pending a network where Tor bootstraps cleanly — see feature 8.*
- **QR friend-add is copy-paste in the web demo**, not a camera scan. Real in-person scanning comes with the mobile apps. **As of feature 18 (0.3.0), only one side pastes**: A shares a code, B pastes it, and B's node auto-delivers its response to A directly over Tor — A never pastes anything back. It's still typed/pasted text either way (a real camera scan is still deferred to iOS), and the auto-connect step needs both onion services reachable — offline degrades cleanly to the original two-way copy-paste ceremony, unchanged underneath.
- **DM forward secrecy is windowed (v0.2).** Devices rotate encryption keys daily and permanently delete retired keys after 7 days: a leaked `keys.json`/backup cannot decrypt DM envelopes older than the window, and an attacker who loses access stops reading new DMs after the next rotation. Still true and stated: a thief with a device that is genuinely *unlocked and running* reads what the app displays — App-lock (feature 13) does not change that boundary, only the idle/asleep/off one (revoke the device instead) — and whole-node-directory theft while unlocked (or on a non-Windows / no-App-lock node) reads cached history exactly as before. One new availability edge, by design: a DM delivered more than rotation+grace after composition (sender partitioned, wrapped to a since-pruned key) arrives permanently unreadable.
- **App-lock is a custody boundary for idle/asleep/off, not for a device in active unlocked use, and is Windows-only.** The secret bundle is sealed at rest under a credential combined with a Windows-DPAPI-sealed device secret (scrypt+HKDF into ChaCha20-Poly1305); the node boots locked and the API 423s every route but unlock/status until the credential is entered, with an escalating throttle and no auto-wipe on a wrong guess. It does **not** protect malware or a shoulder-surfer against a device that is genuinely unlocked and running. OS-suspend detection is a wall-clock-jump heuristic pending a packaged desktop wrapper's real suspend/resume hook, and there is no off-Windows fallback shipped — see feature 13.
- **Deletion is protocol-enforced among honest friends, not DRM.** A malicious peer can retain what it received. This is stated honestly by design, not hidden — and shown in the delete confirm. The out-of-order gossip race (a delete tag arriving before its content silently resurrected DMs/stories/profiles) is fixed and tested for all message kinds (2026-07-03).
- **Unfriend self-deletion is protocol-enforced, not DRM, and delivery-window-bounded.** A modified client can retain a former friend's content, and a screenshot survives everything — same boundary as deletion. A device offline for the full 14-day notice-delivery window never receives it and keeps the content until it does (or until re-friending supersedes it). No block/ban primitive exists; the only path back is the in-person invite ceremony. Resurrection via a mutual friend is structurally blocked (unknown-identity ingest gate), proven by a live three-node test (feature 11).
- **The friend graph is disclosed to authenticated friends** in the sync `have` frame (friend list + peer addresses). That's fan-out control, *not* confidentiality — documented below.
- **Web-only, single-machine demo, now PWA-installable on desktop/Android.** The client installs to a home screen/desktop (manifest + shell-caching service worker, shipped feature 9) but there is still no native mobile client; the installed shell is the same web app, and it still needs its node reachable to show data. **iOS is no longer a PWA target** (see feature 14 / "Next up"): a native iOS app is the committed phone path instead, because a home-screen PWA hits real background/notification limits on iOS Safari that a native app doesn't. The PWA shell still opens fine in iOS Safari today; it just isn't the intended way onto an iPhone going forward.
- **A real (non-demo) desktop node has a branded first-run, a real desktop-app shell, and now a real packaged, distributable `.exe` too — but it's still a friend-test build, not a public release.** `hearth serve` (feature 14) two-phase-bootstraps a fresh dir into a full node via a normal browser tab/window — Create-New-Node or Connect-to-Existing, a one-time wizard, animation, restyled lock screen — completely unchanged. `hearth app` (feature 15, Windows app Phase 1) additionally runs that same node inside a frameless `pywebview` window with custom traffic-light chrome and a Quit-vs-Keep-running close setting. **Feature 17 (Windows packaging Phase 2b)** packages that exact shell into `Kreds.exe` — bundled `tor.exe`, the node running over Tor when frozen, single-instance, freeze-harden. What it is NOT yet: unsigned (SmartScreen warns on first run), no Authenticode/installer/macOS build (public-release follow-ups), and still no real system-tray icon — see "Next up".
- **Auto-update is now a live update channel, mechanism to endpoint — but not yet proven with a real client on another machine.** Feature 16 (Phase 2a) shipped signed-manifest verify-before-apply, a web-asset hot-swap, and core-bundle staging. **Feature 17 (Phase 2b)** completed the chain: a real `.exe` for an update to land on, an on-restart core-swap updater that independently re-verifies and applies a staged core bundle (`hearth/coreupdate.py` + `packaging/launcher.py`), and a real build → sign → publish pipeline to the public `wong001/kreds_updater` feed (`RELEASE.md`). The full chain is proven end-to-end at the logic level (`tests/test_packaging_integration.py`), but **no release has actually been published to the live feed and picked up by a real installed client yet** — that, and the real two-machine Tor test, are the immediate next steps (see "Next up").

## Open design decisions (fold into the concept doc)

Carried from implementation reviews; each is accepted for the current demo but should be recorded and revisited:

- **Offline-overlap delivery (2026-07-09 session, August + Claude).** Two single-device friends with no mutuals who are never online together currently never exchange anything — a topological constraint (the bytes must live on the sender's hardware, the recipient's own hardware, or a third party's; every third party irreducibly learns it holds something). Agreed sequencing: honest delivery state ("reached 3 of 6 devices", rides receipts) → chronic-non-overlap detection nudging an always-on device (local knowledge, zero privacy cost) → home node as the site-recommended setup → **sharded mailbox guardians** for laptop-only circles (k-of-n Shamir shares of sealed, padded envelopes across the recipient's own friends' nodes; ticket-gated deposits; no single guardian holds even the ciphertext — see `docs/superpowers/specs/2026-07-09-sharded-guardians-sketch.md`, needs its own brainstorm → spec) → a central/federated postbox **never** as default (re-creates a provider; undermines both the breach-surface and no-provider-to-order arguments).

- **Entitlement ≠ confidentiality.** Any friend can unilaterally `add_identity(X)` and receive X's signed *plaintext* posts from mutual holders. Per-recipient encryption is the real boundary (out of scope for the current slice). DMs already have this via per-device encryption.
- **Friend-graph disclosure** in the `have` frame (friend list + peer addresses to every authenticated friend). Candidate fix: **hash-based have-exchange** (send `SHA256(identity_pub)` instead of raw) to hide non-mutual friends.
- **Revoke bound** = the revoked device's max-seen seq (the revoker vouches for what it saw; new messages from that device are dead network-wide).
- **DM forward secrecy is windowed (v0.2)** — rotation 24h, grace 7d; finer-grained ratchets rejected for mailbox-latency reasons (spec 2026-07-03). App-lock (feature 13) has since shipped as the device-custody follow-up, Windows-only via DPAPI; a non-Windows or no-App-lock node's custody gap is unchanged. Cleartext ciphertext-blob-hashes in DM payloads are safe *only* under the no-relay/no-group-DM routing invariant.

**Decided against:** screenshot blocking (a DRM problem — mobile `FLAG_SECURE` is defeatable and there's no web/desktop equivalent; consistent with our deletion-honesty stance).
